# 主仓库与子仓库协作约定

## 1. 目的与范围

本手册用于统一管理 `security-capability` 主仓库及 `repositories/` 下的安全能力子仓库，保证各项能力可以独立演进，也能按确定版本组合、复现和交付。

当前子仓库清单以 [`.gitmodules`](.gitmodules) 为准，统一使用 `main` 作为默认协作分支。

## 2. 仓库职责

| 类型 | 仓库 | 职责 |
| --- | --- | --- |
| 主仓库 | `security-capability` | 管理整体目录、子模块清单、自动化脚本和各子仓库的版本引用。 |
| 子仓库 | `repositories/<能力名称>` | 管理某一安全能力的实际代码、文档、配置和测试，独立提交、审查、发布。 |

主仓库并不保存子仓库的文件内容，而是保存每个子仓库对应的提交 SHA（Gitlink）。因此，主仓库某次提交代表一套可复现的能力组合版本。

## 3. 基本原则

1. **子仓库先提交、先推送；主仓库后更新引用、后推送。**
2. 子仓库业务改动必须在对应子仓库内完成提交，不能仅在主仓库执行 `git add`。
3. 更新主仓库中的子模块引用前，确认子仓库提交已推送至远程；否则其他成员无法初始化该引用。
4. 不在 detached HEAD 状态直接开发。先切换到 `main` 或个人功能分支。
5. `.gitmodules` 是子仓库地址和分支的配置来源；新增、删除或迁移子仓库时必须同步更新它。
6. 主仓库中优先显式暂存目标子模块路径，例如 `git add repositories/security-monitoring`，避免使用 `git add .` 误提交无关内容。

## 4. 首次获取项目

```bat
git clone --recurse-submodules git@github.com:SecLeap/security-capability.git
cd security-capability
```

若已经克隆主仓库但未初始化子模块：

```bat
git submodule sync --recursive
git submodule update --init --recursive
```

也可以在主仓库根目录运行：

```bat
submodules.bat update
```

## 5. 子仓库日常开发流程

以 `security-monitoring` 为例：

```bat
cd repositories\security-monitoring
git checkout main
git pull --ff-only origin main

:: 编辑、测试后
git add .
git commit -m "feat: 描述本次能力变更"
git push origin main
```

需要多人协作或较大改动时，应在子仓库创建功能分支并通过合并请求合入 `main`；主仓库只在需要纳入该版本时更新引用。

## 6. 将子仓库新版本纳入主仓库

子仓库提交并推送成功后，在主仓库根目录执行：

```bat
git add repositories\security-monitoring
git commit -m "chore: update security-monitoring submodule"
git push origin main
```

完整顺序如下：

```text
子仓库修改
  → 子仓库提交并推送
  → 主仓库更新该子仓库的 Gitlink 引用
  → 主仓库提交并推送
```

## 7. 同步已有子仓库

在主仓库根目录运行：

```bat
submodules.bat status
submodules.bat update
```

`submodules.bat update` 会拉取已注册子仓库的远程 `main` 分支，并在子仓库引用发生变化时提交、推送主仓库。运行前应确保：

- 当前主仓库和子仓库工作区没有未提交改动；
- 已确认本次需要将所有拉取到的子仓库版本纳入主仓库；
- 本地 Git 已配置身份，且拥有相关仓库的访问与推送权限。

如只想更新一个子仓库，建议使用第 5 节和第 6 节的手动流程，避免把其他子仓库的更新一起带入主仓库。

## 8. 新增或删除子仓库

新增子仓库应使用 Git 子模块命令，而非直接复制目录：

```bat
git submodule add -b main git@github.com:SecLeap/<仓库名>.git repositories/<仓库名>
git add .gitmodules repositories/<仓库名>
git commit -m "chore: add <仓库名> submodule"
git push origin main
```

删除子仓库前应确认不再被依赖，然后依次移除工作区、索引和 Git 元数据；建议由维护者按实际仓库状态执行，避免遗留 `.git/modules` 元数据。

新增或删除后，需同步维护 [`submodules.bat`](submodules.bat) 中的 `REPOS` 清单，确保脚本与 `.gitmodules` 一致。

## 9. 提交信息约定

- 子仓库功能：`feat: ...`
- 子仓库修复：`fix: ...`
- 文档或配置：`docs: ...`、`chore: ...`
- 主仓库更新一个引用：`chore: update <仓库名> submodule`
- 主仓库同步多个引用：`chore: synchronize security capability submodules`

## 10. 常见问题

### 主仓库显示子模块已修改，但没有文件 diff

这是子仓库的提交 SHA 已变化。检查并确认后，在主仓库暂存该子模块路径并提交即可。

### 其他成员初始化子模块失败

通常是主仓库引用了尚未推送的子仓库提交。先在子仓库推送该提交，再更新并推送主仓库引用。

### 子仓库处于 detached HEAD

这是 Git 子模块检出固定提交时的正常状态；需要开发时切换到协作分支：

```bat
cd repositories\<仓库名>
git checkout main
git pull --ff-only origin main
```

### 如何查看当前组合版本

```bat
git submodule status --recursive
```

输出中的每个 SHA 即当前主仓库固定的子仓库版本。
